Flex

Flex is a lexing tool. To use flex, you need to write a .l file and generate the lexing tool you want by doing:

And then you can use the final xxx executable to conduct the task desired. The first line generates the lex.yy.c.

The complete format of a .l file is as follows:

  • We declare global variables and functions in declarion part, which gets copied to the lex.yy.c file.
  • We define tokens/names to be used in the Definition part.
  • We define matching rules in the Rules part. For example: The left part is the pattern of the input described in regular expression. Whenever part of the input matches with the regular expression, the action on the right is triggered. The action is usually written in C. So here, if the input is numbers, a ? gets printed.
    Flex will translate the rules into the function yylex(), which reads in the input file.
  • We add our own subroutines in the last part, which is simply our own C code that gets copied to the lex.yy.c file.

Aside: regular expressions
Some textbooks only cover the most basic regular expressions (*,|), but having a command of other common notations can make it much easier to write a nice pattern.

  • [] The brackets make [abcd] equivalent to (a|b|c|d). You can use - to denote the range in the brackets. E.g. [0-9] means a single digit, and [a-zA-Z] means a single letter.
  • [^] means the complementary set of the char set.
  • * repeat the previous char or bracketed expression for 0 or more times
    • ? repeat the previous char or bracketed expression for 0 time or once
    • + repeat the previous char or bracketed expression for 1 or more times
  • . every char except for the \n char.

Bison

Bison requires us to write a .y file stipulating the productions. Similarly, the .y file follows the format:

The biggest difference from a .l file, is the production part. Here it’s like the SDD in compiling. We follow the pattern to be matched with the actions to be executed when the match happens. The actions are written in C, and should be surrounded by {}.

In definitions however, in addition to declaring tokens here, you can also declare the associativity and precedence. The official document listed %left, %right, %nonassoc, and %precedence as associativity options but i’ve only used the first three. Here %nonassoc specifies no associativity, which means that ‘x op y op z’ is considered a syntax error. As for precedence, the lower the dedfinition is, the higher its precedence.
